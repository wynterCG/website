Tinder-swipe the canvas: building mural gestures

May 4, 2026 · ~7 min read · with Claude Opus 4.7

On this page

The mural canvas in Stridekeep used to be a tap-only target wrapped in Pressable. You tapped a tile to open the placer. That was fine — until I wanted to let players see the other team’s canvas by swiping horizontally on it. The moment I added a swipe, the tap started misfiring on every vertical flick. Time to rewrite.

TL;DR: One PanResponder runs a three-state machine (tap / horizontal pan / vertical no-op) with a Tinder-style live drag and a card-swipe commit. The trickiest fix was a snap-back hijack: if the player re-grabs mid spring-back, the new pan has to be additive on top of the still-running animation, not a fresh start.

The misfire problem

A Pressable doesn’t know what a PanResponder is doing. When the parent intercepts a vertical flick (to scroll the page), the child Pressable still ticks onPress on touch end if the touch moved less than its default tolerance. So pulling the page up to scroll also opened the tile placer on whichever tile your finger started on. Annoying.

The fix isn’t a config flag — it’s replacing Pressable with a gesture that owns the whole touch lifecycle and decides, at the end, whether this was a tap or something else.

Three states, one PanResponder

The CanvasPanel’s single PanResponder classifies every touch into one of three states at release time:

tap
Total movement under TAP_TOLERANCE = 8 dp. Fires the placer for the cell under the start coordinate. No animation. The thing the player intended.
horizontal pan / swipe
Horizontal delta past H_SWIPE_THRESHOLD = 50 dp or a fling with |vx| > 0.4. Commits the own ↔ enemy canvas swap with a card-swipe slide animation.
vertical no-op
If the dominant axis is vertical (e.g. the user is scrolling the page), the responder simply doesn’t claim the gesture. Page scroll wins. No tap fired.
live follow
During a horizontal pan, the canvas follows the finger in real time via onPanMove(dx) — Tinder-style. Release commits or springs back depending on whether the threshold was crossed.

The “vertical drags don’t fire taps” line in the commit message is the one I’m proudest of. The old behaviour was the misfire; the new behaviour is just… correct.

The snap-back hijack

The interesting bug is on re-grab. Say the player drags the canvas 30 dp to the right, lifts (under threshold, so it springs back), then grabs again while the spring is mid-flight. What should happen?

Naive answer: stop the animation, start a new pan from zero. Looks broken — the canvas jumps back to centre instantly before the new drag, which feels like a glitch.

Correct answer: don’t stop the animation. Hijack it. 

// On a fresh grab, if we're mid spring-back:
const current = (panX as any)._value;     // current visual offset
panX.stopAnimation();                     // freeze it where it is
panX.setOffset(current);                  // make that the new origin
panX.setValue(0);                         // delta is now relative to origin

// onPanMove can now do panX.setValue(dx) and the new pan
// rides on top of wherever the spring-back had reached.

The release pass undoes it (flattenOffset() + spring or commit). Claude wrote this pattern on the first try; I never would have, because I’d have reached for .stop() first and stared at the jump.

Don’t simplify: changing the hijack to a naive .stop() is a one-line regression that looks fine in dev (you don’t re-grab during a spring-back when you’re testing features) but is brutally obvious in the hands of a real user. There’s a comment block on this exact temptation in CanvasPanel.tsx now.

Live enemy canvas while you drag

Half the magic of the swipe is what you see during it: as you drag your team’s canvas off-screen, the enemy team’s canvas slides in behind it — and crucially, it’s live. Tiles are being placed in real time on the enemy team’s mural, and you watch them happen.

To make that work, both teams’ canvas data needed to be subscribed simultaneously while CanvasPanel was mounted. Two useCanvasLiveSync hooks (own + enemy) feed the same Supabase realtime channel, and both canvases render through the same pipeline. The swipe is just visual; the data was already there.

Rarity colours, in sync

Same commit, separate system: rarity-coloured names and frames across the live feed, case-reveal modal, inventory grid, and tile picker. Every colour palette in the game has a bucket — common / uncommon / rare / epic / legendary — derived at runtime from tier + dropWeightPct. 

// src/data/colors.ts
export function colorRarity(palette, id) {
  const c = palette[id];
  if (c.tier === 'fine' && c.dropWeightPct < 15) return 'legendary';
  if (c.tier === 'fine' || c.qualityPct < 15)     return 'epic';
  if (c.qualityPct >= 15)                          return 'rare';
  if (c.tier === 'starter' && c.dropWeightPct < 15) return 'uncommon';
  return 'common';
}

A single RARITY_COLOR table is the source of truth for both text tint and frame border tint. <RarityName> renders the colour name in the rarity hue; <RarityFrame> wraps inventory cells with a coloured border. Same data, two surfaces.

The shared legendary blink

Legendary items pulse. The straightforward implementation is: each visible legendary runs its own Animated.loop. The pulse phases would drift between cells. Not catastrophic, but visually ugly — five legendaries blinking out of sync looks broken, not magical.

Solution: a module-level Animated.Value + refcounted Animated.loop inside a useLegendaryBlink hook. Every visible legendary subscribes to the same singleton blink; the loop only runs while at least one is mounted; the refcount unwinds and pauses the loop when the last one unmounts. All legendaries blink in lockstep. Performance bonus: only one animation driver, not N.

UX note worth keeping: rarity-tint the FRAME, not the cell name, in the inventory. Coloured text on the parchment inventory backing was illegible at small sizes. The frame border carries all the rarity signal you need without sacrificing readability of the name itself.

Takeaways

A swipe gesture is rarely just a swipe gesture. The fun part is the slide animation; the unfun parts are the misfires you didn’t anticipate (taps on vertical scroll), the re-grab edge cases (snap-back hijack), and the data dependencies you didn’t realise the visual needed (both teams subscribed live, not just the active one).

The rarity system was a bonus from the same conversation. Once we had the gesture working, “let me show you what an epic-tier tile looks like passing through the picker” became natural, then it spread to four other surfaces in the app. Single source of truth, same data, four visual contracts. The shape of the feature was obvious once we had the gesture as proof-of-concept.
