When the shipped UI beats the spec

Locked Option-D mockups for Stridekeep’s wood-canyon UI shipped on a Wednesday. I installed the release APK on my Samsung A52 that evening, opened it on the couch, and almost immediately wanted to change things. Not because the spec was bad — because seeing it on a phone changed what I thought about it. By the end of the week I had eight intentional divergences from the locked spec, and a problem: the AI agents I was pair-programming with didn’t know about any of them.

TL;DR: When you diverge from a written spec mid-iteration, the AI keeps trying to “fix” back to the spec. The cheap solution is a single written artefact, in the project, that lists every divergence and why. Below: the actual divergences from Stridekeep’s wood-canyon mockups, and the memory file I now maintain so I don’t spend the next month re-litigating them.

The “fix-to-spec” failure mode

Imagine this conversation, two days after the divergence pass:

Me: “Move the case-pending indicator to the right of the sub-tab.”
AI: “The mockup shows the indicator inside the active tab background; I’ll restore that while I’m here.”

The AI isn’t wrong — it’s following the canonical spec it can see. The problem is the canonical spec is no longer canonical. I rejected the “indicator inside the tab” version in real-device review; we now have the indicator floating to the side. But the only place that information lives is in the diff of one commit, which the AI doesn’t naturally re-read every session.

Once I noticed this happening twice in a row, I stopped working on features and wrote a memory file.

What I actually changed

The full list, all shipped, all intentionally different from Option-D:

Canvas backing
Dark slate (colors.bg.overlay), not ocean-blue (colors.canvas.mid). Unfilled tiles at opacity: 0.35. The major 5-cell maroon grid is gone — only the minor per-cell grid remains. Blue + heavy grid felt busy on a small screen.
Frame wood is team-coloured
Outer wood ring of PixelFrame now tracks the viewing team’s accent colour. The mockup’s shared frameWood token was deleted as orphan. Both rings — outer wood and inner hairline — recolour with the team.
Sub-tab background
Inactive #415266 with cream icon/text; active is parchment-hi cream face. marginHorizontal: 3 between tabs, not -1 overlap. The mockup’s overlap made tabs look like a misaligned strip on the A52.
Cases sub-tab indicator
Brass dot pulse for any pending tier (“ready”); brass-rim dot + halo + tab-wide background pulse for fine-tier pending (“rare”). Mounted via a new TabIndicator = 'ready' | 'rare' prop on TabSpec, not the mockup’s in-tab badge.
Bottom-nav active tint
accent.activeGold (#f5c842), not the pale brassRim the mockup used. The pale version read as less saturated than inactive on real hardware — exactly backwards. Cream/gold reads as “you’re here.”
SeasonCard surround
Dark slate fill (colors.bg.overlay) wrapped in a parchment #deaf79 surround (paddingHorizontal: 6, paddingBottom: 6, no top). Mockup recoloured the card itself parchment; cream text lost contrast at typical reading distance.
TopHeader
Stack: shield · “STRIDEKEEP” (no colon, 20pt) + “Season N · MURAL NAME” (11pt) · share / globe / Patreon / Discord (dimmed) / gear. The Buy Me a Coffee icon was dropped during this pass — its two components were deleted as dead code. (Don’t add BMC back; it was rejected for spec reasons.)
CanvasPanel layout
maxCanvasPx = 230 (was 280), SeasonCard stat number 13pt (was 18pt), metaGroup.marginTop = -5 (user-tuned). All so the 3-row LIVE STRIDE FEED fits without scrolling on a 6.5″ phone in portrait.

Each one was decided in front of a real screen, not on a desktop monitor next to a Figma frame. That distinction matters: the mockup is right in the abstract; the device is right in the concrete.

Writing it down for the AI

I now keep two parallel artefacts for any non-trivial UI work:

SKILL.md
The canonical project-level spec (in the repo). Updated whenever a decision becomes load-bearing. §11 has the wood-canyon writeup now, including every divergence above with a sentence on why.
Auto-memory file
A second copy of the same information lives in my Claude auto-memory under wood_canyon_polish_shifts.md. The agent picks it up automatically when working on TopHeader, SeasonCard, SubTabs, CanvasView, CanvasPanel, RootTabs, RarityName, or ColorsPanel — exactly the files the divergences touch.

The auto-memory is the more useful of the two, in practice. It’s pre-loaded into every session so the agent opens with “here are the eight things you might be tempted to revert; don’t.” The SKILL.md is the discoverable artefact for humans (and for future agents that don’t have my memory dir). 

# wood_canyon_polish_shifts.md (excerpt)

- Canvas backing = `colors.bg.overlay` dark slate, NOT
  `colors.canvas.mid` ocean-blue. Unfilled tiles at `opacity: 0.35`.
  Major (5-cell) maroon grid REMOVED — only minor per-cell grid
  remains. Don't reintroduce blue canvas or major grid without
  checking with the user; they explicitly rejected both.

- Mural canvas gestures (single PanResponder). Snap-back hijack
  on re-grab uses `stopAnimation` + `setOffset` so the new pan is
  additive — DO NOT change to a naive `.stop()`.

- Bottom-nav active tint = `colors.accent.activeGold` (#f5c842),
  inactive = `colors.accent.brassLight`. NOT `brassRim` (too pale —
  read as "less saturated than inactive" which the user explicitly
  rejected). Green "Active" sublabel removed.

The pattern that works: every entry leads with the shipped rule, then a why, then the explicit instruction to not revert. The “why” is the load-bearing part — without it, a future agent (or future me) sees only “don’t do X” and starts to second-guess. With it, the rejection makes sense and stays rejected.

Takeaways

Working with an AI pair makes the gap between “my spec” and “the shipped reality” more visible than ever. The AI is constantly cross-referencing what it can see — repo state, mockups, prior commits, memories — and reconciling them. If you let your divergences live only in your head, the AI will quietly “fix” them every time you blink.

So: write them down. Not in a beautifully formatted change-log nobody reads, but in a short, scrappy, opinionated memory file that the AI will pick up on its next session. That file is the cheapest insurance you have against losing your real-device intuitions to a year-old Figma file.
