Place tiles that fly to the canvas

Every PLACE action in Stridekeep used to end with a 600 ms “PLACING… → Placed.” modal hold. Technically fine. Emotionally flat. The whole point of placing a tile is that you did the thing — the moment deserved a beat.

TL;DR: Replaced the placeholder modal with a single celebratory beat — a 140 dp tile of the placed colour appears at screen centre, flies to the destination cell in 380 ms (inOut cubic), and hands off to the existing self-pulse on landing. Took two reverted attempts before Claude and I figured out why Android was off by a status-bar height.

The boring beat

Stridekeep is a team-based idle walking game — real-world steps unlock tiles that you place on a shared medieval canvas. The act of placing should feel like the small celebratory moment it is.

The first version showed a transient modal that read PLACING…, then flipped to Placed. with a checkmark, held for 600 ms, and dismissed. Functional, but the player’s eye was glued to a popup while the actual canvas — the thing they were rewarding themselves on — sat ignored.

Building the fly-to pipeline

The brief I gave Claude: kill the modal hold, replace it with the placed tile materialising at screen centre and animating into its destination cell on the canvas. Sounds easy. The interesting work was wiring this from two completely different call-sites — the canvas TilePicker (you tap an empty cell on the grid) and the inventory PLACE ON CANVAS button (you’re on a different screen entirely).

Transient store field
A pendingTileFlight field on the zustand game store, mirroring how pendingCaseReveal works. Partialize-excluded so it doesn’t persist across reloads.
Two entry points, one overlay
The canvas resolves the destination rect synchronously via a new forwardRef handle on CanvasView. The inventory path hands off with target=null and the canvas resolves the rect once the user lands on Home.
Animation envelope
380 ms, easing Easing.bezier(0.4, 0, 0.2, 1), animating both translate and scale in lockstep. Native-driver-safe so it stays smooth even when the JS thread is busy.
Hand-off on landing
The realtime echo + existing self-pulse take over the moment the flight completes. No extra coordination needed — the overlay just unmounts.

The conversation with Claude on this was very “you’re the systems architect, I’m the UX critic”. It proposed a clean pipeline; I pushed back on a few details (the original landing transition was too crisp; I wanted a barely-perceptible squash on touchdown). Most of an afternoon, but the structure stuck on the first real iteration.

The Android Y-offset bug

And then iOS worked perfectly. Android was off by exactly 31.7 dp — the height of the status bar — every single time.

We’d already reverted two attempts on this. The third time, I asked Claude to just sit with the symptom for a while before suggesting a fix. The clue was in the consistency: the offset was always the same, always positive, always equal to the status bar inset. It wasn’t a race; it was a coordinate-system mismatch.

What was happening: on Android with edge-to-edge mode enabled (which Stridekeep uses), measureInWindow on a view inside a Modal returns Y values offset by the status bar height from where root-level absolute-positioned overlays render. Even with statusBarTranslucent set, the two frames disagree.

The fix was anticlimactic — once we knew what we were looking at: 

// TileFlightOverlay.tsx
const insets = useSafeAreaInsets();

// On Android, measureInWindow() inside a Modal reports Y values offset
// by the status bar height vs. where the root-level overlay renders.
// Compensate by adding the safe-area top/left to the resolved rect.
const targetRect = {
  x: rawTarget.x + insets.left,
  y: rawTarget.y + insets.top,
  width: rawTarget.width,
  height: rawTarget.height,
};

Two lines of compensation. Documented in a comment block in TileFlightOverlay.tsx so neither I nor any future AI session re-discovers it as “mysterious 32-pixel drift on Android”. This memory now also lives in my Claude auto-memory so it survives sessions — when a future agent tries to debug the same symptom, the answer is one grep away.

Race conditions Claude caught

Once the happy path worked, the superpowers:code-reviewer agent went through and flagged a handful of race conditions I would absolutely have missed. The good ones:

Rapid double-place
If a user taps PLACE twice quickly, the second flight could mount on top of an in-flight first one using the same React key. Fixed by deriving a stable per-flight key on FlightModal so each fresh mount gets its own animated values.
Orphan flights
Inventory hand-offs cleared themselves when the user landed on Home and the canvas resolved the rect — but if the user never reached Home (e.g. backgrounded the app mid-PLACE), the flight could sit pending forever. Added a 1500 ms ORPHAN_TIMEOUT_MS safety net.
RPC-failure cleanup
Canvas commits go through Supabase RPC. If the RPC failed AFTER the flight started, the error alert would fire over a still-running animation. Now we clear the flight state before Alert.alert.
Memoised onComplete
The flight overlay’s onComplete callback was being recreated each parent render, which triggered useEffect resets inside the overlay. useCallback fixed it.

None of these were going to bite me in dev. All four would have shipped to a real player at some point. This is the part of vibe coding that surprises people: the AI writes faster, but the bigger leverage is that it also reviews differently — a second pair of eyes that doesn’t care about your ego or your weekend.

Zoom + halo on top

Once the fly-to was solid, I layered two more effects on top of it for the celebratory beat:

Camera zoom anchored on the destination cell

1.0 → ZOOM_PEAK 1.6 → 1.0 over 380 ms in, 600 ms hold, 400 ms out. The math reduces to tx = dx * (1 - scale) where dx is the cell-centre minus canvas-centre in canvas-local coordinates — so the cell stays at its same screen position while the surrounding canvas scales up around it. Camera leans in on the placed cell without yanking it elsewhere.

Self-placement glow halo

Two stacked semi-transparent rings (inner cellPx × 2.4 at 55% alpha, outer cellPx × 4.2 at ~25% alpha) layered under the existing self-pulse. A single glowOpacity Animated.Value drives both via interpolation. Peaks at 400 ms, decays to 0 over the remaining 1600 ms. Only fires for your own placements — teammate placements stay unchanged.

Tiny visual gift: the canvas overflows the wooden frame at peak zoom — the matte surround doesn’t overflow:hidden, so a hairline of canvas leans forward through the frame on the celebratory beat. I didn’t plan it. Claude flagged it as a potential bug. I kept it.

Takeaways

A celebratory micro-animation is rarely just a celebratory micro-animation. The fun part is tile motion. The unfun parts — coordinate-system mismatches across native modals, race conditions in concurrent transitions, cleanup when the network fails halfway — are where most of the time goes.

Working with Claude on this kept the unfun parts cheap. I described the symptom and we narrowed it together; the code-review agent caught four things I would have shipped. The flight beat itself was the easy part.
