Breakout Roguelike is a Godot 4 game where the bricks are alive — monsters that move, shoot, and have HP — and every level you pick from 3 upgrades. It’s been a sandbox for me to figure out how I want to work with an AI pair on a long-lived game project. The methodology that emerged surprised me: every commit starts with a phase number.

TL;DR: Each phase is one focused conversation with Claude that ends in a single commit. Phase numbers go phase4_00, phase5a_00, phase5b_00… You can read the project history as a sequence of design questions and the AI’s implementation answers. The naming convention is the methodology, not the bookkeeping.

A weird hybrid

Imagine Breakout, except the bricks are 8 different monster types with movement patterns (drift, zigzag, orbit, charge), shooting patterns (straight, aimed, spread, burst), and HP that scales 15% per level. Now imagine that after each level you pick one of three random upgrades from a 20-card pool — burn DOT, chain lightning, paddle laser, lifesteal, thorns. That’s the shape.

Designing a hybrid like this isn’t a single sit-down task. It’s a hundred small design decisions — “does the paddle move freely in 2D or just left-right?”, “do upgrade cards cost tokens?”, “should monsters drop physical pickups or just count down on death?” — each of which produces a chunk of code plus a chunk of art plus a chunk of tuning.

One phase, one commit, one conversation

I had no plan to be this methodical. The phase names emerged because, sitting down to write commit messages, I kept needing to label what kind of change this was. The label that stuck wasn’t feature-name (too granular) or area-name (too vague) — it was the iteration: 

phase4_00      Upgrade system + UpgradeManager autoload
phase5a_00     Formation data system (Cell/Data/LevelPool)
phase5b_00     Formation Editor plugin + axis-based bounce
phase5c_00     Per-cell size scaling
phase5d_00     30-level balance + 8 monsters
phase6a_00     Monster movement (drift/zigzag/orbit/charge)
phase6b_00     35 symmetric formation templates
phase6c_00     Monster shooting + 15-col formations
phase6d_00     Small arc paddle with 2D movement
phase6e_00     20 upgrades + damage numbers + laser visual
phase6f_00     Token economy
phase6g_00     Balance pass + mixed formations

The _00 suffix is there in case a phase ends up needing a hotfix follow-up (phase6e_01 for instance), but in practice almost every phase is one commit. The cadence is: start a conversation, decide the goal, build it, review it, commit it. Repeat.

One conversation rule: each phase lives entirely inside one chat session with Claude. If a thread is getting too long or veering off into a different feature, I close it, commit what we have, and start a fresh one for the next phase. Context-window discipline = scope discipline.

The actual phase log

Looking back, the phases group into design waves. Each wave is roughly a week and starts with an architectural question:

WaveQuestionPhases
Phase 4 “How do upgrades work between levels?” 4_00 only — UpgradeManager singleton, 12 starter upgrades, pick-1-of-3 overlay.
Phase 5 “How do I author levels without hand-placing 30 layouts?” 5a (data model) → 5b (visual editor + axis-based bounce) → 5c (per-cell size scaling) → 5d (30-level balance).
Phase 6 “How do bricks become characters, not just HP sponges?” 6a (movement) → 6b (formations) → 6c (shooting) → 6d (paddle 2D) → 6e (20 upgrades) → 6f (tokens) → 6g (balance).

The question is the unit of work. The phases inside a wave are the architectural answer broken into implementable pieces. Each phase commit body is its own small dev log — a list of bullets describing what landed, including the Godot-specific gotchas that surfaced (private vars on Projectile, read-only array issues with resource creation, font scaling for blocks with per-cell size, the ball-below-paddle “bounces toward the paddle” bug fix).

Why it works

Phase numbers are a contract

When I open a fresh session with the AI, the conversation often starts with “we’re in phase 6f. Phase 6e shipped the laser upgrade and damage numbers; we’re moving to a token economy now.” That single sentence anchors the entire chat. The AI knows what level of polish is appropriate (phase 6 is feature-additive, not refactoring), what data structures already exist (UpgradeData, FormationData, MonsterData), and what the prior phase decided.

Commit messages double as design notes

The commit body for phase6f reads like a feature spec, because it is the feature spec — every decision the AI and I made during that conversation got captured by the time we committed: 

phase6f_00: Token economy — physical pickups, upgrade costs, monster values

- Monsters drop golden diamond tokens on death with gravity physics
- Paddle collects tokens on contact, missed tokens disappear at bottom
- Each monster has editable token_value (Slime=1, Demon=5, Dragon=8)
- Single token per monster showing value number
- Upgrade selection: 3 rounds per level, cards cost 1/2/3 tokens
- Unaffordable cards greyed out + unclickable (mouse_filter=IGNORE)
- Skip button to save tokens, tokens carry over between levels
- Token counter in top-right during gameplay, hidden during upgrade picks
- HP bar + token UI hidden during upgrade selection (clean overlay)

Six months from now, when I forget why mouse_filter is set to IGNORE on unaffordable cards, the answer is in the phase that introduced them. The git history is the documentation.

The code review is a phase ritual

Every phase ends with the superpowers:code-reviewer agent doing a pass. Phase 5b had 11 fixes in the code-review block alone — read-only array bugs, type mismatches (thorns_damage float → int), a piercing-reset bug that fired every frame instead of only after a bounce, an upgrade-card cleanup that should have run immediately instead of via queue_free. Without the review pass, half of those would be live bugs people would only find by playtesting.

Takeaways

Vibe coding isn’t just chat-with-an-AI. It’s a workflow that scales with how you frame the work. Naming phases gives the AI a shared frame; ending each phase with a commit gives it a clean handoff for the next session; ending with a code-review pass keeps the bug debt low. None of these are new ideas — the news is how cheaply you can implement them when the AI does most of the typing.

On Breakout Roguelike, six weeks of evenings produced 20 upgrades, 8 monsters with movement & shooting, 35 symmetric formation templates, a visual formation editor plugin, a token economy, and 30 hand-tuned levels. Phase numbers were the spine.
