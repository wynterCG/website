A clean, modular idle → battle pipeline. This post shows what existed, what the battle slice adds, and why the pieces click together without drama.

TL;DR: Single-player, data-driven, with a battle screen that swaps in, runs a tidy turn loop, awards XP, and hands control back to the idle layer.

Pre-Battle Foundation

Before combat, the project already had a solid backbone: a central controller managing UI and input; a modular trio for economy (Purr), progression (Leveling), and attributes (Stats); and a resource loop (Purr Power) pacing bigger moments. UI was dynamic and performance-friendly, with dual live previews and packaged-build fixes in place.

Economy (“Purr”)
Active & passive gain, growth curves, and upgrade hooks.
Leveling
XP → Levels → Points, with clean thresholds and pacing.
Stats & Attributes
Long-term values feeding derived combat numbers; player point spend.
UI & Previews
Dynamic lists, two live scene captures, throttled for performance.
Production polish
Stable references & texture handling so package ≈ editor.

The Battle Slice

The slice mounts cleanly on top of that foundation. Encounters start from the main UI, freeze current stats into a per-fight snapshot, alternate turns with readable logs.

Flow
Start → Player turn ↔ Enemy turn → AutoBattle.
Enemy data
Defined in a table for fast authoring (ID, name, HP, ATK, XP).
Snapshot
Freeze stats at fight start so mid-fight tweaks don’t drift numbers.
Combat feel
Hit/crit/dodge basics, simple mitigation, fair flee checks.
UI
Scroll battle log; fight button .
Events
Combat broadcasts; UI listens. No tight coupling.

Player Experience & Tuning

Early upgrades nudge reliability and pace (accuracy, survivability, resource gain). Randomness is seeded once per session for stable testing while still feeling fresh.

Good Practices & Optimization

Under the hood, the project follows a few habits that keep it fast, legible, and easy to extend. This is the practical checklist I used while building the slice.

Data-first, then visuals

Enemies, upgrades, and long-term stats live in data assets/tables. That means balance work, content additions, and variant testing don’t require touching gameplay graphs. Visuals (widgets, previews) read from data and react to events.

Modularity & separation

Each major concern sits in its own module: economy, leveling, stats, and combat. The UI doesn’t know how numbers are produced; it just displays what it’s told. This lowers risk when changing any one system and makes testing straightforward.

Blueprint Interfaces

UI talks to gameplay via interface messages (e.g., “start fight”, “pet button”,"upgrade buttons", "upgrade stats"). No hard references needed, which prevents circular dependencies and makes swapping controllers/characters safe. Interfaces also keep input handling tidy.

Event Dispatchers

Gameplay emits simple, one-way events: state changes, HP updates, and log lines. Listeners receive the values they need and update in place. This keeps widgets lightweight and avoids polling or tight coupling.

Naming tip: Use “OnThingChanged(Value…)” for dispatchers and keep them focused. If the data itself is the point of the event (HP, State, Log), include it in the payload.

RNG you can trust (and replay)

A single Random Stream is seeded once per session. All chances (hit, crit, flee) draw from it. For debugging, switch to a fixed seed and you’ll reproduce the exact same sequence—perfect for tracking “impossible” streaks.

  • Session seed: random at startup for fresh play; fixed in debug for repeatability.
  • From-Stream rolls: use the “from Stream” random nodes so the sequence advances consistently.
  • Auditability: log the seed + first few rolls when debugging; makes bug reports actionable.

Optimization that matters

  • No wasteful ticks: timers for cadence; widgets update only on events.
  • SceneCapture throttle: previews update on a controlled interval, not every frame.
  • Lightweight UI: small log item widget, autoscroll, and no expensive per-frame bindings.
  • Packaged parity: stable references for spawned objects; texture priority quirks handled.
