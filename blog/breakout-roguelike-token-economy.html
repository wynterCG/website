Tokens and a formation editor

Two phases I want to put side-by-side: 5b built a visual editor plugin for Godot so I could author boss formations without touching code, and 6f built a token economy where killing monsters drops golden diamonds you spend on upgrades. Different surfaces, same vibe-coding pattern: describe the system in plain English, let the AI sketch a structure, iterate from there.

TL;DR: The formation editor is an in-Godot plugin with paint tools, a level-pool manager, a cell inspector, and a draggable split view. The token economy gives each monster a token_value, drops physical pickups with gravity, and gates upgrade cards at 1/2/3 tokens each. Both phases ended with a code-review pass that caught a bunch of subtle Godot pattern bugs.

The formation editor

Phase 5a introduced the data model: FormationCell, FormationData, and LevelPool resources. A formation is a grid of cells, each cell points at a monster role (front/tank/support/elite). A level pool references several formations and picks one when the level loads. Clean. Authoring it by editing .tres files by hand was painful.

So phase 5b built the editor. The brief I gave Claude was four bullets:

Visual grid
Click a cell to place / clear / change a monster role. Paint tools so I can drag a brush.
Template dropdown
100 starter patterns (T-shapes, walls, V-formations) to begin from, then customise.
Arena preview
A second panel showing the blocks at their real in-game positions, not the editor grid coords.
Level pool manager
Add / remove / reorder formations across the 30 level pools without leaving the editor.

What came out was a docked Godot plugin with two split panes — grid editor on the left, arena preview on the right — joined by a draggable divider. Cells got their own inspector for fine tuning (width/height scale 0.5x–3x, monster role enum, per-cell size scaling). Saving rewrites the underlying FormationData.tres.

The non-obvious lesson from this phase came from the code-review pass: read-only array issues when modifying nested resources. Godot’s Resource arrays are shared by reference until you duplicate them, so the “add cell” button was occasionally mutating the template formation instead of the current edit. The fix was to always create a fresh FormationData on edit, never mutate the template. Claude caught this one; I’d have shipped it.

Axis-based bounce

Same phase, separate problem: ball bounces. Godot’s built-in Vector2.bounce(normal) works fine until the ball clips a block corner — the surface normal is ambiguous, and the ball reverses direction in a way that doesn’t feel right (sometimes it bounces backwards into the player, sometimes through the next block).

Replaced it with axis-based reflection: compare the ball’s position to the block’s centre, force the direction away on whichever axis is dominant. The diff for walls and paddles followed the same shape: 

# Blocks: bounce away on dominant axis
var diff = ball.global_position - block.global_position
if abs(diff.x) > abs(diff.y):
    ball.velocity.x = sign(diff.x) * abs(ball.velocity.x)
else:
    ball.velocity.y = sign(diff.y) * abs(ball.velocity.y)

# Walls: normal axis only, force away
if normal.x != 0:
    ball.velocity.x = sign(normal.x) * abs(ball.velocity.x)
else:
    ball.velocity.y = sign(normal.y) * abs(ball.velocity.y)

# Paddle: ball below = force up; ball above = arc-angle math
if ball.global_position.y > paddle.global_position.y:
    ball.velocity.y = -abs(ball.velocity.y)
else:
    apply_arc_angle()

Plain. Predictable. Removes the corner/edge bounce reversal bug. The “ball below paddle bounces toward paddle” class of bug — a beloved Breakout footgun — goes away by construction. Looking up the original bounce maths was a thing Claude was much faster at than me, but the value-add was actually framing the problem as “same shape on three surfaces” so we wrote it once and reused it.

The token economy

Two months later, phase 6f gave the game an economy. Killing a monster now drops a golden diamond token with gravity physics. The paddle catches the diamond by walking under it. Miss it, it falls off the bottom of the screen. Tokens accumulate, and you spend them on upgrades between levels.

Per-monster value
Each monster type has an editable token_value export — Slime = 1, Goblin = 2, Knight = 3, Demon = 5, Dragon = 8. The number floats on the token sprite so the player can see at a glance which kills are worth chasing.
Physical pickup
On death, the monster instantiates a Token CharacterBody2D with downward velocity and standard gravity. The paddle has a pickup area; entering it grants the value and queues the token for free. Missed tokens despawn at screen bottom.
UI choreography
Token counter top-right during gameplay. Hidden during the upgrade-selection overlay so the picker reads clean. HP bar also hidden — the only thing visible during a pick is the three upgrade cards.
Skip + carry-over
Skip button on the upgrade picker saves all your tokens for next level. Lets the player save up for a 3-cost epic instead of being forced to buy something every round.

Cost discipline on upgrade cards

The interesting design question wasn’t how to do tokens — it’s what they should gate. The answer that landed: each upgrade card costs 1, 2, or 3 tokens, and there are three picks per level. If you’re flush with tokens, you can grab three cards a level; if you’re skint, you can skip-skip-skip and bank for an epic next time.

The technical detail that took the most iteration: how does the player know a card is too expensive to afford? The version that shipped:

Visual: greyed out
Unaffordable cards desaturate and dim. Hover state doesn’t fire. The price tag is still legible so the player understands “I need one more token to afford this”.
Behavioural: unclickable
Setting mouse_filter = MOUSE_FILTER_IGNORE means the card eats no input — clicks pass through to whatever’s underneath (which is nothing, so they just bounce off). No “you can’t afford this” error popup. The card simply doesn’t respond.

Two-line implementation, half an hour of conversation to get there. Most of that half-hour was Claude and me discussing what to do when the player runs out of tokens entirely — auto-skip? auto-end-level? force a free common? The answer we settled on: show the picker anyway, with all cards greyed, and an emphasised Skip button. It’s rare but it happens, and the affordance for it is the existing Skip button, just visually highlighted.

Design rule that stuck: never lock the player out of the picker UI. Even when they can’t afford anything, show the cards so they can read what they could have bought, and decide whether to chase tokens harder next level.

Takeaways

The formation editor was a tool I built for myself; the token economy was a feature I built for the player. Both came out of single phase-conversations with the AI: describe the goal, sketch the shape, iterate until it lands, run the code-review pass, commit.

The thing I’d tell my past self about vibe coding: build the editor before the content. Phase 5b’s editor unblocked phases 5c, 5d, 6b, 6c, and 6g — every formation-tuning pass after it was 5× faster because I could see what I was authoring. Three hours building a tool saved fifteen hours editing .tres files by hand.
